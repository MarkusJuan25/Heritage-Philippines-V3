## D-011 — Authentication library selection

- **Status:** Accepted
- **Date accepted:** July 10, 2026
- **Decision:** Better Auth, with its official Prisma/PostgreSQL adapter (`better-auth/adapters/prisma`), is the authentication and session library for Heritage Philippines V3. This fulfills ADR-001's deferred "Select a vetted authentication/security library after checking its official documentation" follow-up action (ADR-001, "Immediate, pre-Phase-1-scaffolding").
- **Rationale:** Evaluated against Lucia and Auth.js (NextAuth). Lucia was deprecated by its maintainer in March 2025 and is no longer distributed as an installable library (repositioned as an educational resource) — disqualified as "maintained." Auth.js's Credentials provider is architecturally hardcoded to JWT-only sessions and cannot use real database-backed sessions without unsupported workarounds, conflicting with ADR-001's stated preference for "database-backed session state" and this project's rule against inventing custom session mechanics. Better Auth supports credentials-based login and database-backed sessions natively through its official Prisma adapter, hashes passwords internally (scrypt) rather than requiring custom cryptography, and supports disabling public self-service sign-up (`emailAndPassword.disableSignUp`) while keeping sign-in enabled — required to satisfy blueprint Section 7's invitation-only-signup rule. As of September 2025 the Auth.js/NextAuth maintainers themselves joined the Better Auth project and now recommend it for new projects. It requires no serverless-specific hosting platform, remaining compatible with Hostinger-managed VPS-class Node.js hosting (ADR-001).
- **Constraint:** This selects a library within ADR-001's already-approved architecture (Next.js, Prisma, PostgreSQL, cookie-based sessions) and does not alter the accepted technology stack. It does not implement custom cryptography, password hashing, or session-token primitives, consistent with ADR-001's security requirements.
- **Effect:** Implements the technology choice underlying the Phase 1 "Establish authentication/session foundation" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. Final MFA/SSO and any additional auth providers remain open per blueprint Section 16.2.

## D-012 — Better Auth admin plugin evaluated and rejected for staff management

- **Status:** Accepted
- **Date accepted:** July 13, 2026
- **Decision:** The installed Better Auth admin plugin (`better-auth@1.6.23`, `better-auth/plugins/admin`) was evaluated as a candidate implementation for the Phase 1 "staff account, role, and assignment management" checklist item and rejected. Heritage Philippines V3 will instead implement a custom, audited staff-management service (Approach B) built on the project's existing Prisma client, role guards (`src/lib/auth/roles.ts`, `authorize.ts`, `guards.ts`), and only the non-plugin Better Auth server primitives already in use (e.g., `hashPassword` from `better-auth/crypto`, as `prisma/seed.ts` already does). The `admin` plugin must not be registered in `src/lib/auth/auth.ts`, and no `/api/auth/admin/*` route may exist.
- **Rationale:** Direct inspection of the installed package's compiled source (`dist/plugins/admin/{admin,routes,schema,types,has-permission}.mjs`) found three conflicts with Heritage's requirements: (1) the plugin's endpoints (`set-role`, `create-user`, `ban-user`, `remove-user`, etc.) are unconditionally mounted on the same catch-all handler this project's `/api/auth/[...all]` route already forwards everything to — there is no supported way to use the plugin's server APIs internally without also exposing those endpoints over HTTP, authorized only by the plugin's own parallel `adminRoles`/`ac` permission config rather than this project's fixed six-role model; (2) the plugin redefines `user.role` as an unconstrained string (`type: "string"`, not required), a different shape from this project's own six-value literal-union `role` field, and its `setRole` endpoint has no self-action or "last remaining admin" protection; (3) its mutation endpoints call Better Auth's internal adapter directly with no exposed transaction hook, so an `AuditLog` write can never be atomic with the mutation it records, and `removeUser` is a hard delete rather than the deactivate/reactivate model the blueprint calls for.
- **Constraint:** Heritage retains its fixed, blueprint-approved six-role `Role` enum and keeps permissions expressed in code rather than adopting the plugin's own role/permission system. The staff-management service (implemented in a later checkpoint) must perform staff account creation, role changes, and deactivation/reactivation as direct, atomic Prisma operations (mutation + `AuditLog` insert in one transaction), not through any Better Auth admin-plugin endpoint.
- **Effect:** Closes the Approach A vs. B vs. C evaluation for this checklist item in favor of Approach B. Direct credential provisioning (a System Administrator setting a new staff account's initial password) must be covered by an integration test in the staff-management-service checkpoint that verifies the created account can actually authenticate through the normal Better Auth sign-in path — not merely that a database row was written correctly.
- **Implementation status update (July 21, 2026):** The staff-management service and its four `SYSTEM_ADMINISTRATOR`-only API routes are implemented and unit-tested. The integration requirement in D-012 is satisfied by `apps/web/src/features/staff/service.integration.test.ts`: all three tests passed against the disposable local PostgreSQL database `heritage_v3_test` using the real `createStaffAccount`, `deactivateStaffAccount`, and Better Auth `auth.api.signInEmail` paths. This verifies sign-in using both generated and caller-supplied passwords, plus sign-in rejection and session revocation after deactivation. `heritage_v3_dev` was not used; staging and production remain unverified.

## D-013 — Server-generated booking reference format

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `Booking.bookingReference` (blueprint Section 13.4) is generated exclusively by the server as the literal prefix `HPB-` followed by 20 uppercase hexadecimal characters (e.g. `HPB-7A31F09C45D2E86B104F`), derived from 10 cryptographically secure random bytes via Node's built-in `crypto.randomBytes` — no new dependency is introduced. It is never supplied or overridable by a client: `POST /api/bookings`'s request schema (`apps/web/src/features/bookings/schemas.ts`) has no `bookingReference` field at all, and the service layer (`apps/web/src/features/bookings/service.ts`'s `generateBookingReference`) is the only place this value is produced.
- **Rationale:** Blueprint Section 13.4 names "Booking reference" as a Booking field but specifies no format, sequence, or generation rule anywhere in the blueprint, this decisions log, or the task board — this was an explicitly open item identified during the Booking Schema Foundation checkpoint's implementation reconnaissance. A random, non-sequential, non-year-based identifier avoids leaking booking volume or timing through the reference itself and avoids the coordination cost a sequential counter would require under concurrent creation. `Booking.bookingReference`'s existing database `@unique` constraint (migration `20260716020734_booking_schema_foundation`) remains the final safeguard regardless of generation method; the booking-creation service additionally retries with a freshly generated reference, within a small bounded retry policy, on the rare event of a collision.
- **Constraint:** Sequential or human-readable (e.g., year-prefixed, incrementing) booking references are deferred unless management later requires them — this decision does not preclude revisiting the format, but any change must go through this same decision process, not be introduced ad hoc in a later checkpoint.
- **Effect:** Resolves the previously open "booking reference generation" item identified during the Booking Schema Foundation checkpoint. Implemented as part of the booking workflow enforcement checkpoint (explicit Booking creation, staff-only list/read, initial `BookingStatusHistory`, and the `BOOKING_CREATED` audit entry) — status-transition enforcement and Booking-specific staff-assignment endpoints remain separate, not-yet-implemented checkpoints.

## D-014 — Booking status-transition matrix, authorization, and concurrency

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `PUT /api/bookings/[id]/status` enforces the following fixed, service-layer transition matrix over `BookingStatus` (blueprint Section 13.4's ten values), independent of any database constraint:
  - `DRAFT` → `PENDING_CONFIRMATION`, `CANCELLED`
  - `PENDING_CONFIRMATION` → `CONFIRMED`, `CANCELLED`
  - `CONFIRMED` → `IN_PREPARATION`, `CANCELLED`
  - `IN_PREPARATION` → `DOCUMENTS_REQUIRED`, `VISA_PROCESSING`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `DOCUMENTS_REQUIRED` → `VISA_PROCESSING`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `VISA_PROCESSING` → `DOCUMENTS_REQUIRED`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `READY_FOR_TRAVEL` → `IN_PROGRESS`, `CANCELLED`
  - `IN_PROGRESS` → `COMPLETED`, `CANCELLED`
  - `COMPLETED` → no outgoing transitions (terminal)
  - `CANCELLED` → no outgoing transitions (terminal)

  Authorization mirrors Booking creation/read exactly: `ADMIN_MANAGER` (unconditional) and `TRAVEL_CONSULTANT` (only through the existing active-Client-assignment check, `features/bookings/repository.ts`'s `clientAssignmentFilter` — not a new Booking-specific assignment mechanism, which remains unimplemented). `SYSTEM_ADMINISTRATOR`, `FINANCE_ACCOUNTING`, `VISA_DOCUMENTATION`, and `CLIENT` are not authorized to change Booking status in this checkpoint.

  Concurrency: the caller supplies `expectedStatus` alongside `newStatus`; the service reads the Booking's actual current status inside a `runSerializableWithRetry` transaction and rejects with `BOOKING_CONFLICT` (409, refresh-and-retry message) if the current status no longer matches `expectedStatus` — an optimistic-concurrency check layered on top of (not a replacement for) the transaction's own serializable-conflict retry. A request whose `newStatus` already equals the current status is a no-op (no history row, no audit entry, no write) — required by `booking_status_history_status_changed`'s existing CHECK constraint, which forbids a history row where `previousStatus` equals `newStatus`.

  `DOCUMENTS_REQUIRED` and `VISA_PROCESSING` are treated as coarse overall booking-stage indicators only, not as a model of the detailed, genuinely parallel document/visa workflows blueprint Section 8 describes ("payments, documents, visa processing... may all progress in parallel rather than strictly in sequence"). `Booking.status` remains a single scalar reflecting the booking's overall stage; the future `Document` and `VisaCase` entities (blueprint Sections 14.7, 14.8) own their own detailed, independently-progressing status fields once implemented. No schema change accompanies this decision.
- **Rationale:** Blueprint Section 13.4 lists the ten status values but defines no transition graph or role authority over them, and Section 8 explicitly warns that the underlying workflows are parallel, not sequential — modeling that fully would require the not-yet-implemented `Document`/`VisaCase` entities, which is out of scope for this checkpoint. The matrix above is a deliberately conservative starting point (forward progression plus universal cancellation from any non-terminal state, since no blueprint text restricts when cancellation is allowed) rather than an attempt to fully resolve the parallel-workflow tension now. Reusing active-Client-assignment (rather than introducing Booking-specific `StaffAssignment` targets, which the schema supports but no service/route yet populates) keeps this checkpoint additive and avoids authorizing against an assignment type nothing can currently create.
- **Constraint:** This matrix, and the two-role authorization set, may be revised by a later decision once `Document`/`VisaCase` are implemented and the parallel-workflow question is revisited; until then, no additional role or transition should be added ad hoc outside this decision. A `reason` field for status changes is explicitly deferred, not adopted, in this checkpoint.
- **Effect:** Implements the "Booking status-transition enforcement" checkpoint referenced in `docs/HERITAGE_V3_TASK_BOARD.md`. Booking-specific staff-assignment management and any Document/VisaCase-driven status model remain separate, not-yet-implemented checkpoints.

## D-015 — Booking-specific staff-assignment enforcement

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `PUT /api/bookings/[id]/assignment` lets an Admin / Manager set or replace the Travel Consultant assignment for a specific Booking, extending the existing `features/assignments` shared implementation (`TargetKind` now includes `'BOOKING'` alongside `'LEAD'`/`'CLIENT'`) rather than introducing a separate Booking-assignment service. Booking assignment is **independent of the underlying Client assignment**: the assignee is never required to already hold the Booking's Client's active assignment, and no cascade exists in either direction — ending, replacing, or creating a Client assignment never touches an existing Booking assignment, and vice versa. This checkpoint **does not affect** Booking read, list, or status-transition authorization: `features/bookings/repository.ts`'s `clientAssignmentFilter` and the D-014 status-transition authorization rules are unchanged and continue to gate Travel Consultant access via active Client assignment only. This checkpoint is **`PUT`-only** — no `DELETE`/removal-without-replacement endpoint exists for Booking assignment; ending a Booking assignment without an immediate replacement is deferred, not decided against permanently.
- **Rationale:** Blueprint Section 4.2 explicitly grants Admin / Manager the capability to "manage staff accounts' operational role assignments and lead/client/booking assignments," naming Booking directly — the same explicit basis Lead and Client assignment already rely on. The underlying `StaffAssignment.bookingId` column, its three-way `leadId`/`clientId`/`bookingId` XOR CHECK constraint, and its partial unique active-Booking-assignment index were already added by the Booking Schema Foundation checkpoint (migration `20260716020734_booking_schema_foundation`) but never used by any service or route until now — extending the existing generic `setAssignment`/`endAssignment` control flow (idempotent same-assignee replay, `REASON_REQUIRED` only when replacing an active assignment, atomic end-old/create-new/audit inside `runSerializableWithRetry`) maximizes reuse of already-reviewed, already-tested logic rather than duplicating it. Leaving Booking read/status authorization on Client-level assignment avoids conflating two separate concerns — "who currently works this Booking" (new, additive) and "who may read or transition this Booking" (existing, unchanged) — until a future decision, if any, deliberately links them.
- **Constraint:** Whether a Booking assignment must eventually require, or should remain independent of, the Client assignment; whether Booking read/status authorization should ever incorporate Booking-level assignment; and whether assignment removal without replacement should be added, are all left open for a future decision — none is silently resolved here. No Prisma schema or migration change was required or made for this checkpoint.
- **Effect:** Implements the "Booking-specific staff-assignment enforcement" checkpoint referenced in `docs/HERITAGE_V3_TASK_BOARD.md`. `features/bookings/repository.ts`, its status-transition service, and their routes remain completely unchanged.

## D-016 — Conversation Schema Foundation: schema shape, read-state model, and migration-history remediation

- **Status:** Accepted
- **Date accepted:** July 17, 2026
- **Decision:** The Conversation Schema Foundation schema-only checkpoint (blueprint Sections 4.2–4.6, 9.1, 10, 10.1, 10.2, 10.3, 13.9, 14.3) adds `ConversationCategory` (the eight closed categories in Section 10.2, adopted directly), `MessageVisibility` (`CLIENT_VISIBLE`/`INTERNAL_NOTE`, Section 10.1), `ConversationParticipantRole` (`CLIENT`/`TRAVEL_CONSULTANT`/`FINANCE_ACCOUNTING`/`VISA_DOCUMENTATION` — deliberately excluding `ADMIN_MANAGER`, since Section 10.1 treats Admin/Manager's full visibility as a role-wide grant, not thread participation), `Conversation`, `ConversationParticipant`, `Message`, and `MessageAttachment`. Key shape decisions:
  - `Conversation.clientId` is a required direct FK to `Client` (mirroring `Lead`/`Proposal`/`Booking`), since a Conversation may have no related Booking or Proposal at all.
  - A Conversation may reference **at most one** related record in this checkpoint — an optional `relatedBookingId` or an optional `relatedProposalId`, never both (enforced by CHECK, see below). A `relatedVisaCaseId` is not modeled: VisaCase does not exist anywhere in this schema yet, matching its existing deferral as a `StaffAssignment` target.
  - `ConversationParticipant` uses two mutually exclusive identity paths — `staffUserId` (User) for `TRAVEL_CONSULTANT`/`FINANCE_ACCOUNTING`/`VISA_DOCUMENTATION` and `clientProfileId` (ClientProfile) for `CLIENT` — tied together by one CHECK constraint per participant row. `joinedAt`/`removedAt` preserve membership history (mirroring `StaffAssignment.endedAt`) rather than deleting a row on removal, and a person may rejoin after removal.
  - Read state (Section 10.1's "read status") is modeled as **`ConversationParticipant.lastReadAt`** — a per-participant timestamp, not a shared boolean on `Message` — because a Conversation may have more than two participants and a single flag cannot represent "read" correctly for all of them at once. A `Message` is considered read for a given participant when the message's `createdAt` is at or before that participant's `lastReadAt`; the future service layer owns updating `lastReadAt` and resolving timestamp/ordering edge cases. `lastReadAt` is read-state storage only — it never itself grants or implies authorization.
  - `Message` uses the same two-path author split (`authorStaffUserId`/`authorClientProfileId`), and this checkpoint adds one content-level database restriction beyond path exclusivity: a client-authored message can never have `visibility = INTERNAL_NOTE`.
  - `MessageAttachment` stores only the minimal metadata the backend-mediated storage boundary needs (`storageKey` unique, `fileName`, `mimeType`, `fileSizeBytes`) — never a public URL.
  - No Conversation status/lifecycle field, subject/title field, or `lastMessageAt` denormalization is added — none is defined anywhere in blueprint Section 10.
  - Conversation/message **authorization**, and the cross-table client-consistency invariants (`relatedBooking.clientId`/`relatedProposal.clientId` must equal `Conversation.clientId`; a `CLIENT` participant must belong to `Conversation.clientId`), are explicitly **not implemented** in this checkpoint. A future enforcement checkpoint must account for all of: Admin/Manager's full, unconditional visibility into every Conversation; a CLIENT actor's ownership of the Conversation's Client (via ClientProfile) together with explicit `ConversationParticipant` membership; a Travel Consultant's active Client/Booking assignment; and explicit `ConversationParticipant` membership for Finance/Visa Documentation actors, who have no assignment-based path onto a Conversation at all. Conversation authorization must never be described or implemented as "participant rows alone." Server-side exclusion of `INTERNAL_NOTE` messages from every client-facing query is likewise deferred to that same future checkpoint.
- **Rationale:** Directly follows Section 10.1's documented structure (thread/participants/messages/attachments, client-visible-vs-internal-note) and Section 10.2's closed category list, without inventing any undocumented field (status, title, `lastMessageAt`). The per-participant `lastReadAt` model was chosen over a shared Message-level boolean specifically because Section 10.1 describes more than two possible participants per thread (Travel Consultant, optional Finance, optional Visa Documentation, plus the Client), which a shared flag cannot represent correctly. The staff/client split-identity pattern on both `ConversationParticipant` and `Message` reuses this schema's established `StaffAssignment`/`ProposalAcceptance` precedent rather than inventing a new attribution shape.
- **Migration-history note:** This checkpoint's schema is split across two migrations rather than one, because of an implementation-session incident: `prisma migrate dev --create-only` was run a second time to confirm the first run's exit code, and Prisma's documented behavior of always applying any already-pending unapplied migration at the start of a `migrate dev` invocation (regardless of `--create-only` on that specific run) caused the first-generated migration — at that point containing only the auto-generated tables/enums/standard indexes/foreign keys, before any hand-written constraint had been added — to be applied to the local `heritage_v3_dev` database. Inspection confirmed: all four new tables were empty at the time (no data was ever at risk), no object outside the four new tables referenced them, and no pre-existing table's structure was altered. Rather than editing the already-applied migration file after the fact (which would desync it from the checksum Prisma already recorded against it in `_prisma_migrations`, and `prisma migrate resolve --rolled-back` is not a valid operation against a migration that completed successfully — it did not fail), the remediation was forward-only: migration `20260717010445_conversation_schema_foundation` (tables/enums/standard indexes/FKs) was left byte-for-byte unchanged; the second, previously-empty migration was renamed to `20260717010520_conversation_schema_constraints`, populated with every hand-written PostgreSQL CHECK constraint and partial unique index this checkpoint requires, and reviewed. Following that review, **both migrations were successfully applied to local `heritage_v3_dev` on July 17, 2026** via `prisma migrate dev` (the base migration was already applied from the incident above; only the constraints migration was newly applied in this step). `prisma migrate status` subsequently reported all 8 migrations applied and the schema up to date. A read-only PostgreSQL catalog check confirmed all four hand-written CHECK constraints (`conversation_related_record_exclusive`, `conversation_participant_identity_role_match`, `message_author_path_exclusive`, `message_client_author_not_internal_note`) are present and validated, and both partial unique indexes (`conversation_participant_active_staff_key`, `conversation_participant_active_client_profile_key`) are present, unique, and valid with the correct predicates. This application is confirmed for local `heritage_v3_dev` only — application to staging or production has not occurred as part of this checkpoint and must be verified independently before either environment is considered current.
- **Constraint:** No later checkpoint should combine these two migrations or edit either one's SQL after the fact. Conversation/message authorization, cross-client consistency checks, VisaCase as a third related-record type, email notification delivery (Section 10.3), and attachment malware scanning/signed-URL issuance (Section 12.3) all remain open for a future checkpoint, not resolved here. Conversation-creation authorization (who may open a Conversation, and whether a client may self-initiate one for particular categories) and the exact policy governing who may author a Message (CLIENT ownership plus active participation; explicit active participation for Finance/Visa Documentation; Travel Consultant assignment with the participant-row question still open; and whether Admin/Manager may post by role alone without participant membership) are both likewise left open — authorship foreign keys alone must never be treated as proof of authorization by any future implementation.
- **Effect:** Implements the Conversation Schema Foundation slice of the Phase 1 "Implement database schema and reviewed migrations from the approved Phase 0 entity plan" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. Payment, Document, and VisaCase schema remain unimplemented, so that checklist item is not marked complete by this decision.

## D-017 — Visa Case Schema Foundation: MVP granularity, status list, and Stage A schema shape

- **Status:** Accepted
- **Date accepted:** July 17, 2026
- **Decision:** The Visa Case Schema Foundation checkpoint (blueprint Sections 4.5, 4.7, 5, 5.1, 5.2, 8, 13.8, 14.1, 14.8) is implemented in stages; this entry covers **Stage A (schema-and-decision-documentation only — `schema.prisma` and this decisions-log entry, with no migration created or applied)**. Approved:
  - **One VisaCase per Booking for the MVP.** A Booking may have zero or one VisaCase; a VisaCase belongs to exactly one Booking. Enforced by `VisaCase.bookingId String @unique` — a plain unique constraint, not a partial one, since this checkpoint has no historical/superseded VisaCase concept for a partial index to distinguish. Multiple historical or simultaneous VisaCases per Booking, and traveler-level VisaCase modeling (one case per traveler on a multi-traveler Booking), are both explicitly out of scope.
  - **The exact nine-value `VisaCaseStatus` enum:** `OPENED`, `REQUIREMENTS_PENDING`, `UNDER_REVIEW`, `ADDITIONAL_INFORMATION_REQUIRED`, `READY_FOR_SUBMISSION`, `SUBMITTED`, `APPROVED`, `REJECTED`, `WITHDRAWN`, with `APPROVED`, `REJECTED`, and `WITHDRAWN` as terminal. This is the one status enum in this schema whose values are set by explicit decision rather than adopted directly from blueprint text — blueprint Section 13.8 states "the detailed visa status list will be finalized alongside implementation planning" and Section 16.2 separately lists it as an open decision; this entry is that finalization. No transition matrix is adopted or implied by the enum's declaration order or by which values are marked terminal — transition enforcement (including enforcing terminality) is explicitly deferred to a later service/API checkpoint, mirroring how `BookingStatus`'s transition matrix was deferred to its own later, separate D-014 checkpoint.
  - **Direct `Client` and `Booking` relations on `VisaCase`**, mirroring this schema's established `Lead.clientId`/`Proposal.clientId`/`Booking.clientId`/`Conversation.clientId` convention, both `onDelete: Restrict`. `VisaCase.clientId` must always equal the Client reachable through `VisaCase.booking.clientId` — a cross-record rule Prisma cannot enforce, deferred to mandatory service-layer verification at write time, the same way `Booking.clientId` vs. its `ProposalVersion.proposal.clientId` is already deferred. No duplicated or derived-column schema logic was used to attempt to solve this.
  - **VisaCase as a StaffAssignment target at the schema level only.** `StaffAssignment.visaCaseId` (nullable) is added in Stage A; the existing three-target XOR CHECK constraint and the corresponding partial unique active-assignment index are **not yet altered in any database** — a later migration-generating stage must extend `staff_assignment_lead_xor_client_xor_booking` to a four-target `staff_assignment_lead_xor_client_xor_booking_xor_visa_case` (`CHECK (num_nonnulls("leadId", "clientId", "bookingId", "visaCaseId") = 1)`) and add `staff_assignment_active_visa_case_key` (`WHERE "endedAt" IS NULL AND "visaCaseId" IS NOT NULL`). `features/assignments`' `TargetKind` union, service, audit, and authorization code are untouched in Stage A and remain so until a separate, later assignment-enforcement checkpoint, mirroring D-015's precedent for Booking exactly.
  - **VisaCase as the third optional Conversation related-record target at the schema level only.** `Conversation.relatedVisaCaseId` (nullable) is added in Stage A; the existing two-column `conversation_related_record_exclusive` CHECK is **not yet altered in any database** — a later migration-generating stage must replace it with `CHECK (num_nonnulls("relatedBookingId", "relatedProposalId", "relatedVisaCaseId") <= 1)`.
  - **Every `VisaStatusHistory` row requires an attributable staff actor under D-017's MVP policy.** `changedByUserId` is required, not nullable, matching `BookingStatusHistory` rather than `LeadStatusHistory`. Blueprint Section 5.1 permits deliberate creation through an explicit staff or system action; D-017 deliberately narrows the current MVP schema to staff-attributed creation and status history. A future automated or actorless VisaCase transition would require a superseding decision and schema change.
  - **`VisaRequirement` and all Document entities (`Document`, `DocumentRequirement`, `DocumentReview`) are deferred**, not modeled even as a placeholder. `VisaRequirement`'s one-sentence blueprint description (Section 14.8: "defines what is required for a specific VisaCase (documents, forms, deadlines)") overlaps structurally with `DocumentRequirement` (Section 14.7: "defines which document categories are required for a given Booking or VisaCase"), and neither `Document` nor `DocumentRequirement` exists anywhere in this schema yet.
  - **Assignment-service wiring, VisaCase authorization, the status-transition matrix, API routes, admin dashboard and client portal UI, and notifications are all deferred** to later, separate checkpoints — none is implemented, implied, or assumed by Stage A's schema shape.
  - **No migration has been generated or applied during Stage A.** No `prisma migrate` command was run. `schema.prisma`'s new and changed models, fields, relations, and doc comments do not correspond to any database table, column, constraint, or index anywhere — local, staging, or production — until a later stage generates and a still-later stage applies the migration.
- **Rationale:** Blueprint Section 5.2's conceptual diagram presents VisaCase as a singular conditional relationship (`Booking --(conditional, only if required)--> Visa Case`) and Section 8 confirms a visa case is conditional and never automatic, but neither section independently defines complete cardinality — the blueprint does not itself state whether a Booking could ever have more than one VisaCase over time. D-017 is the explicit decision establishing zero or one VisaCase per Booking for the MVP, chosen because it avoids adding an unapproved Traveler entity or storing loosely modeled applicant identity data during the MVP. The plain-unique (not partial-unique) `bookingId` constraint reflects that this checkpoint has no "current vs. superseded" VisaCase concept to distinguish, unlike `ProposalVersion` or `StaffAssignment`. Deferring `VisaRequirement` and Document entities avoids prematurely designing overlapping structures before the document domain (`Document`/`DocumentRequirement`) is implemented, consistent with this schema's established caution about not modeling a shape that would likely need reworking (e.g. TourPackage deferred entirely). Staging the checkpoint (schema-and-decision-documentation-only Stage A, migration-generation Stage B, application in a still-later stage) is a direct, deliberate response to the D-016 migration-history incident: keeping schema editing, migration generation, and migration application as separate, individually-reviewable steps reduces the chance of a repeat, and Stage A explicitly documents that Stage B must invoke `prisma migrate dev --create-only` exactly once.
- **Constraint:** The four-target `StaffAssignment` XOR CHECK, its `visaCaseId` partial unique index, the three-target `Conversation` related-record CHECK, and the `visa_status_history_status_changed` CHECK are all specified here as requirements for the future migration-generating stage but must not be added to any migration file during Stage A. Who may create or change a VisaCase (beyond Visa Documentation Staff, blueprint Section 4.5), whether Admin/Manager's Section 4.2 "full visibility" extends to VisaCase edit capability, and how VisaCase-level and Booking-level `StaffAssignment` should interact are all left open, not resolved here.

## D-018 — Document Schema Foundation: single-entity versioning, derived status, and requirement/context boundaries

- **Status:** Accepted
- **Date accepted:** July 17, 2026
- **Decision:** The Document Schema Foundation checkpoint (blueprint Sections 4.2–4.6, 12.1–12.3, 13.1, 13.3, 13.4, 13.7, 14.7) is implemented in stages; this entry covers **Stage A (schema-and-decision-documentation only — `schema.prisma` and this decisions-log entry, with no migration created or applied)**. Approved:
  - **Single-entity versioning, no `DocumentVersion` model.** Each `Document` row represents one immutable uploaded file/version. A replacement creates a new `Document` row linked back to the original through `previousDocumentId`; the replaced row is retained, never deleted or mutated (blueprint Section 12.2: "a replaced document is retained, not deleted"). Unlike Proposal/ProposalVersion (blueprint Section 14.4, which explicitly names both), Section 14.7 names only `Document` itself with "version" as one of its own fields — no separate `DocumentVersion` entity is introduced.
  - **`Document.status` is not stored; no `DocumentStatus` enum exists.** Operational status is derived by the future service/repository layer in this exact priority order: (a) if another `Document` references the current row through `previousDocumentId` — equivalently, the current row's `replacementDocument` reverse relation is non-null — the current (older) row is SUPERSEDED/REPLACED, regardless of any `DocumentReview` it may also carry; (b) otherwise, if the current `Document` has no `DocumentReview` rows at all, it is PENDING_REVIEW; (c) otherwise, its status is the latest `DocumentReview.outcome`, with "latest" selected deterministically using `reviewedAt DESC, createdAt DESC, id DESC` — a service/repository ordering rule, not a new database constraint or index requirement. `DocumentReviewOutcome` (`APPROVED`, `REJECTED`, `REPLACEMENT_REQUESTED`) is the only closed enum this checkpoint adds, taken directly from blueprint Section 14.7's own closed three-value list.
  - **`documentRequirementId` is nullable.** Proactive or ad-hoc uploads are permitted; a document counts toward a formally requested requirement only when explicitly linked. Nothing in blueprint Section 12.1 mandates a requirement-first workflow.
  - **Context targeting.** `Document.clientId` is always required (direct FK, mirroring this schema's established convention). A `Document` may additionally target at most one of `bookingId`/`visaCaseId` (never both) — a client-only Document with both null is permitted. `DocumentRequirement` targets exactly one of `bookingId`/`visaCaseId` (never neither, never both) — read from blueprint Section 12.1's "linked to a DocumentRequirement that defines what is required for a given booking **or** visa case."
  - **Direct-context ownership invariants (cross-row, service-layer only — not expressible as a Prisma/PostgreSQL CHECK across related tables).** If `Document.bookingId` is non-null, the referenced `Booking.clientId` must equal the Document's own `clientId`; if `Document.visaCaseId` is non-null, the referenced `VisaCase.clientId` must equal the Document's own `clientId`. Both checks apply unconditionally — including when `documentRequirementId` is null — because a Document's context ownership must hold regardless of whether it happens to be requirement-linked. When `documentRequirementId` is set, the linked `DocumentRequirement`'s own target (`bookingId` or `visaCaseId`) must exactly match the Document's target of the same kind and resolve to the same Client as the Document's `clientId`; a requirement-linked Document must never be client-only (a non-null `documentRequirementId` requires exactly one of `bookingId`/`visaCaseId` also non-null). All of these are deferred to mandatory transactional service-layer verification at write time, the same way this schema already defers the analogous `Booking.clientId`/`VisaCase.clientId` cross-row checks elsewhere.
  - **Uploader-identity invariants (cross-row, service-layer only — no database role/status column or CHECK is added).** The `document_uploader_path_exclusive` CHECK proves only that exactly one of `uploadedByStaffUserId`/`uploadedByClientProfileId` is populated; it proves nothing about the referenced identity's role, account status, or ownership. When `uploadedByClientProfileId` is used, the referenced `ClientProfile.clientId` must equal the Document's own `clientId`. `uploadedByStaffUserId` may only ever represent an authenticated, active, operationally-authorized staff actor at write time — a CLIENT-role `User` must never be recorded via `uploadedByStaffUserId`. Exact per-role upload capability boundaries remain deferred to the future Document service/API checkpoint, matching this schema's existing deferral of analogous role-boundary enforcement elsewhere.
  - **Replacement integrity.** `previousDocumentId` is nullable and `@unique`, preventing two Documents from sharing one predecessor (a replacement-chain fork). Services must ensure a replacement retains the same client/context/requirement/category lineage as the row it replaces and increments `version` by exactly one. Long-cycle prevention (proving a replacement chain terminates) remains a service-layer invariant; the `@unique` constraint alone cannot prove it.
  - **Review reason.** A non-blank `reason` is required on `DocumentReview` when `outcome` is `REJECTED` or `REPLACEMENT_REQUESTED`, per `.claude/rules/admin-dashboard.md`'s explicit rule that rejecting a document requires a logged reason; `APPROVED` may carry an optional reason/note. The later migration must enforce this with `CHECK ("outcome" NOT IN ('REJECTED', 'REPLACEMENT_REQUESTED') OR COALESCE(length(btrim("reason")), 0) > 0)` — the `COALESCE` guard is required, not a plain `IS NOT NULL` check or a bare `length(btrim(reason)) > 0`, because PostgreSQL treats a CHECK expression that evaluates to `NULL` as satisfied, and a bare `length(btrim(NULL))` comparison would silently evaluate to `NULL` (and therefore pass) for a `REJECTED`/`REPLACEMENT_REQUESTED` row with no reason at all.
  - **Reviewer-identity invariants (cross-row, service-layer only — no new enum, role field, or database constraint is added).** `reviewedByUserId` being a required, non-nullable FK to `User` proves only that a `User` row is referenced, not that User's role or account status. It is a service-layer invariant, not a database constraint, that `reviewedByUserId` must resolve to an authenticated, active, operationally-authorized staff actor at write time; a CLIENT-role `User` must never create, or be recorded as, a `DocumentReview`. The exact capability boundary between Admin/Manager, Travel Consultant, and Visa Documentation Staff for which review outcomes each may record remains deferred to the future Document service/API checkpoint, consistent with this schema's existing deferral of analogous role-boundary enforcement elsewhere — that deferral concerns only which staff roles may review, not whether clients may; clients reviewing documents is out of scope under every reading of blueprint Section 4.
  - **VisaRequirement boundary.** `DocumentRequirement` owns uploadable document-category obligations for both Bookings and VisaCases. Future `VisaRequirement` work (still fully deferred) may model broader visa procedural requirements, forms, and deadlines, but must not duplicate `DocumentRequirement`'s uploadable-document responsibility — resolving the overlap D-017 itself already flagged between blueprint Sections 14.7 and 14.8.
  - **Access/download logging deferred.** `ActivityLog`-based access and download tracking (blueprint Section 12.3: "access and download audit logs for every document") is deferred to Phase 4 implementation and must exist before document access/download endpoints are considered complete. The existing `AuditLog` model remains scoped to sensitive/security-relevant staff actions and must not be repurposed as the routine download-event store.
  - **Storage metadata boundary.** Only the storage-metadata fields already established by `MessageAttachment` are used: `storageKey` (an internal private-storage key, never a permanent public URL), `fileName`, `mimeType`, `fileSizeBytes`. No storage-provider/bucket identifier, checksum, malware-scan-status, or document-content field is added in this checkpoint — none is blueprint-required, and ADR-001 places actual malware-scanning/storage-provider configuration in Phase 4, alongside the Documents feature.
  - **No migration has been generated or applied during Stage A.** No `prisma migrate` command was run. `schema.prisma`'s new and changed models, fields, relations, and doc comments do not correspond to any database table, column, constraint, or index anywhere — local, staging, or production — until a later stage generates and a still-later stage applies the migration.
- **Rationale:** Blueprint Section 14.7 names exactly three entities (`Document`, `DocumentRequirement`, `DocumentReview`) with no separate version or submission entity, and describes `Document` itself as carrying "category, version, and status" as its own fields — the single-entity-with-version reading follows that literal text rather than inventing a fourth entity blueprint doesn't name. Deriving status rather than storing it mirrors this schema's established aversion to redundant derived state (e.g. `ProposalVersion`'s accepted/locked state). The nullable `documentRequirementId` and permissive client-only-Document shape avoid over-constraining a workflow blueprint Section 4.6 describes clients as free to use ("Upload, view, and track their own documents") without a stated requirement-first precondition. Deferring `ActivityLog`, storage-provider/checksum fields, and `VisaRequirement` avoids modeling entities or columns this checkpoint has no blueprint-sourced shape for, consistent with this schema's established caution against premature, likely-to-be-reworked design (the same reasoning D-017 applied to `VisaRequirement`/Document entities before this checkpoint existed to resolve it).
- **Constraint:** The `document_requirement_target_exclusive`, `document_context_exclusive`, `document_uploader_path_exclusive`, `document_version_positive`, `document_previous_not_self`, and `document_review_reason_required` CHECK constraints are all specified here as requirements for the future migration-generating stage but must not be added to any migration file during Stage A. Whether Admin/Manager's Section 4.2 "full visibility" extends to document review/approval capability, and how Travel Consultant's "excluding visa-category documents" boundary (Section 4.3) is enforced against `Document.category` at the service layer, are both left open, not resolved here.
- **Effect:** Implements Stage A (schema-and-decision-documentation only) of the Document Schema Foundation slice of the Phase 1 "Implement database schema and reviewed migrations from the approved Phase 0 entity plan" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. That checklist item remains unchecked: Payment schema remains unimplemented, and Document itself has no migration yet. `docs/HERITAGE_V3_TASK_BOARD.md` is not updated by this entry — its own update is a later step.

A `prisma migrate status` check on July 20, 2026 confirmed this migration is applied to the local `heritage_v3_dev` database — 10 migrations found, database schema up to date. ## D-019 — Payment Schema Foundation: currency-per-Booking, allocation model, refund correction, and Stage A schema shape

- **Status:** Accepted
- **Date accepted:** July 20, 2026
- **Decision:** The Payment Schema Foundation checkpoint (blueprint Sections 4.2-4.7, 5.1-5.2, 11.1-11.7, 13.4, 13.6, 14.5, 14.6) is implemented in stages; this entry covers **Stage A (schema-and-decision-documentation only — `schema.prisma` and this decisions-log entry, with no migration created or applied)**. Approved:
  - **Currency refinement** (supersedes D-002's PHP-only-MVP wording without deleting it — see D-002's own Refinement note above). Each Booking selects exactly one currency (`Booking.currencyCode`); PHP and USD are supported examples, not an exhaustive list; every financial amount within that Booking uses that Booking's selected currency; mixed-currency and exchange-rate handling within a single Booking are deferred.
  - **`Booking.totalAmount`/`currencyCode` are added, both nullable.** Neither field can be required from day one: `createBookingSchema` accepts only `proposalVersionId` today, and `ProposalVersion` carries no pricing content yet, so no source exists to populate either field at Booking-creation time; a `NOT NULL` column without a default would also fail to apply against any existing Booking row. Both fields must exist (be non-null) before a `PaymentPlan` may be created for that Booking, and neither may change once a `PaymentPlan` or `Payment` exists for it — both service-layer invariants, not database-enforced (no trigger mechanism is used anywhere in this schema).
  - **One `PaymentPlan` per `Booking` for the MVP** (`PaymentPlan.bookingId @unique`, a plain unique index, mirroring `VisaCase.bookingId`'s own precedent). `PaymentPlan` does not duplicate `totalAmount`, `currencyCode`, or a separate deposit-amount field.
  - **Deposit represented as `Installment.isDeposit Boolean`**, the smallest justified field for this checkpoint's evidence — not a new enum. A migration-only partial unique index enforces at most one deposit Installment per plan; the future service layer must additionally enforce that a deposit, when present, is `sequenceNumber = 1`. Whether any given plan requires a deposit at all follows that plan's own approved booking terms — blueprint Section 11.4 does not state every plan must include one, so this checkpoint does not assume universality.
  - **No stored Installment status and no waiver fields.** Installment payment state (Scheduled/Due/Overdue/Partially Paid/Paid) is fully derived from allocations, refunds, due date, and amount by the future service layer. No approved requirement in this repository supports installment waivers (blueprint Section 11.6 lists "Waived" only as one illustrative example within a still-unfinalized list), so no `waivedAt`/`waivedByStaffUserId`/`waivedReason` field is added.
  - **`Payment` targets `Booking` and `Client` directly, never `installmentId`.** Which Installment(s) a Payment covers is recorded through the new `PaymentAllocation` join model instead, supporting one Payment covering multiple Installments, one Installment receiving multiple Payments, and partial payments (blueprint Section 11.4) — a shape a single nullable `installmentId` column cannot express. `Payment` carries no inline confirm/reject/reverse/refund timestamp, actor, or reason field; every transition is recorded on the new append-only `PaymentStatusHistory` instead, mirroring `BookingStatusHistory`'s established convention, so `Payment.status` can never drift out of sync with a duplicated field set.
  - **`PaymentAllocation.amount` is always positive, never signed.** This model is append-only: a misallocation is corrected by writing a `PaymentAllocationReversal` (a complete, one-time reversal of exactly one allocation — `paymentAllocationId @unique`) and then creating new, separate positive `PaymentAllocation` rows, never by mutating or deleting the original. A `PaymentAllocation` with any existing `PaymentRefundAllocation` rows against it must not be reversed unless the service can prove the reversal preserves every refund-allocation invariant; default service policy is to reject that operation. A confirmed amount left unallocated to any Installment is unapplied Booking credit and is never automatically applied to a future Installment.
  - **`PaymentRefund` is a separate, append-only record** — a row exists only once money has actually been returned, so this checkpoint models no "pending refund" state. Multiple partial refunds against one Payment are supported. A Payment stays `CONFIRMED` while partially refunded and becomes `REFUNDED` only once cumulative refunds equal its full amount; a Payment with any refund must never transition to `REVERSED` (refunded and reversed are mutually exclusive); refund totals must never exceed the original Payment amount. `PaymentRefundAllocation` links a `PaymentRefund` to the specific `PaymentAllocation` row(s) it reduces (positive amount, `@@unique([paymentRefundId, paymentAllocationId])`), preserving exactly which Installment's balance reopens; any refunded amount not linked to an allocation is a refund from unapplied Booking credit.
  - **Corrected net-contribution formula, fixing an earlier draft's defect.** An earlier draft computed "gross confirmed payment total" using only `status = CONFIRMED` while separately subtracting all refunds regardless of status, so a fully refunded Payment (whose status becomes `REFUNDED`, no longer `CONFIRMED`) produced a *negative* net contribution instead of exactly zero. The corrected, approved formula is: `net contribution of Payment P = CASE WHEN P.status IN (CONFIRMED, REFUNDED) THEN P.amount - SUM(PaymentRefund.amount for P) ELSE 0 END`; `net confirmed amount paid = SUM(net contribution of every Payment under the Booking)`; `remaining Booking balance = max(Booking.totalAmount - net confirmed amount paid, 0)`; `overall Booking overpayment = max(net confirmed amount paid - Booking.totalAmount, 0)`; `net active allocation (Installment Y) = SUM(PaymentAllocation.amount WHERE installmentId = Y AND payment.status IN (CONFIRMED, REFUNDED) AND no PaymentAllocationReversal exists for that allocation) - SUM(PaymentRefundAllocation.amount for those allocations)`; `unapplied Booking credit = net confirmed amount paid - SUM(net active allocation across every Installment)`; `outstanding Installment amount = max(Installment.amount - net active allocation, 0)`; `next payment due = earliest dueDate among installments with outstanding amount > 0`. The service layer must guarantee unapplied Booking credit and every net active allocation can never become negative, by actively verifying it at write time. This formula also required a narrow clarification to `docs/HERITAGE_V3_PROJECT_BLUEPRINT.md`'s Section 11.1/11.2 (made alongside this entry) — the blueprint's "Refunded" status-effect row was worded at the whole-payment-status level, before partial refunds were an approved feature; the amendment clarifies that a partially refunded Payment stays `CONFIRMED` and reduces the balance only by its refunded portion.
  - **`PaymentStatus` is the exact seven-value closed list from blueprint Section 11.2's table** (`PENDING`, `CONFIRMED`, `REJECTED`, `CANCELLED`, `FAILED`, `REFUNDED`, `REVERSED`). No `VOIDED` or `PARTIALLY_REFUNDED` value is added.
  - **`PaymentStatusHistory`** mirrors `BookingStatusHistory` exactly (append-only, required actor, nullable `previousStatus` only for the initial row). Every real transition after the initial `PENDING` record requires a non-blank `reason` (standardized, service-generated reasons are acceptable) — stricter than `document_review_reason_required`'s outcome-conditional pattern. `idempotencyKey` is required specifically for transitions to `CONFIRMED`, `REVERSED`, or `REFUNDED`. `Payment.status`, this history row, and the required `AuditLog` entry must be written in one future transaction.
  - **Idempotency keys** are added to `PaymentAllocation`, `PaymentAllocationReversal`, `PaymentRefund`, and (conditionally) `PaymentStatusHistory` — each required and unique, since none of these writes has a natural uniqueness to detect a retry by (unlike `Booking.proposalVersionId`). The future service layer must check for an existing row by the caller-supplied key before inserting and return that result on a retry, exactly mirroring `bookings/service.ts`'s existing handling of a `bookingReference`/`proposalVersionId` unique-constraint race.
  - **`Receipt` is an immutable historical snapshot** — `amount` and `currencyCode` captured at issuance, no `updatedAt` field, one per Payment (`paymentId @unique`, already anticipated by `.claude/rules/database-security.md`'s "one confirmed Receipt per confirmed Payment" example), issued only for a validly confirmed Payment, and never deleted or mutated by a later reversal or refund. No receipt-number prefix format is enforced — only uniqueness.
  - **`Document.paymentId` (nullable) reuses the existing Document/DocumentReview machinery** for client-submitted payment-proof evidence, rather than a parallel review model. The existing `document_context_exclusive` CHECK is widened from two columns to three (`num_nonnulls("bookingId", "visaCaseId", "paymentId") <= 1`), following the exact drop-and-re-add pattern already used for the Visa Case Schema Foundation's widening. `DocumentReview`'s role-boundary doc comment is narrowly amended: Finance/Accounting may review a Document only when its `paymentId` is non-null — a structural check, not a `category` string match, since `Document.category` remains free-text with no closed list or CHECK constraint (blueprint Section 12.1); no `PAYMENT_PROOF` category constant or CHECK is invented. Approving/rejecting such a Document and the corresponding `Payment.status` transition must occur atomically in a future service layer.
  - **No `StaffAssignment` target for Payment or PaymentPlan.** Payment authorization derives entirely from the existing Booking-level assignment; no new target column is added.
  - **No migration has been generated or applied during Stage A.** No `prisma migrate` command was run. `schema.prisma`'s new and changed models, fields, relations, and doc comments do not correspond to any database table, column, constraint, or index anywhere — local, staging, or production — until a later stage generates and a still-later stage applies the migration.
- **Rationale:** The corrected net-contribution formula, one-PaymentPlan-per-Booking, positive-only append-only allocations with a dedicated reversal model, and a dedicated `PaymentStatusHistory` all follow this schema's established preference for append-only history and derived state over mutable or duplicated fields (`Document.status`, `ProposalVersion`'s accepted/locked state, `BookingStatusHistory`). Modeling `Payment` against `Booking` directly, with allocation as a separate join, follows blueprint Section 14.6's own "against an Installment or Booking" phrasing literally rather than forcing a single FK to carry two different meanings. Deferring `PaymentAdjustment`, the refund documentation type, and the post-approval Installment amendment workflow avoids guessing at a shape this repository has no approved requirement for yet, consistent with this schema's established caution — the same reasoning D-017 and D-018 each applied to entities/fields they had no blueprint-sourced shape for.
- **Constraint:** The `booking_total_amount_positive`, `booking_financials_pairing`, `booking_currency_code_format`, `payment_plan_approval_pairing`, `installment_amount_positive`, `installment_active_deposit_key` (partial unique index), `payment_amount_positive`, `payment_status_history_status_changed`, `payment_status_history_reason_required`, `payment_status_history_idempotency_key_required`, `payment_allocation_amount_positive`, `payment_allocation_reversal_reason_required`, `payment_refund_amount_positive`, `payment_refund_reason_required`, `payment_refund_allocation_amount_positive`, `receipt_amount_positive`, `receipt_currency_code_format`, and the widened `document_context_exclusive` constraints are all specified here as requirements for the future migration-generating stage but must not be added to any migration file during Stage A. Whether `Installment` corrections after `PaymentPlan` approval require their own append-only pattern, and the exact `PaymentAdjustment` shape, are both left open, not resolved here.
- **Deferred:** `PaymentAdjustment` is not modeled until its exact balance effect is approved. The refund documentation type (a refund reference, credit note, or equivalent client-facing document, blueprint Section 11.5/16.2) remains open — `PaymentRefund` is a structured internal financial record only, not automatically that document. A post-approval Installment amendment/correction workflow is left open. Payment-gateway/processor fields, mixed-currency/exchange-rate handling within one Booking, an installment-status enum, and automatic credit-to-installment allocation are all explicitly excluded from this checkpoint.

## D-020 — Notification Schema Foundation: minimal in-app alert shape and Stage A schema

- **Status:** Accepted
- **Date accepted:** July 21, 2026
- **Decision:** The Notification Schema Foundation checkpoint (blueprint Section 14.9) is implemented in stages; this entry covers **Stage A (schema-and-decision-documentation only — schema.prisma and this decisions-log entry, with no migration created or applied)**. Approved:
  - **Notification is implemented as a minimal Phase 1 schema foundation now, rather than explicitly deferred to Phase 5.** Blueprint Section 15.3 lists Notifications as included in MVP, and D-019's own implementation-status update identified Notification as the sole remaining blueprint Section 14 entity with neither an implementation nor a documented deferral — this entry closes that specific gap.
  - **Each Notification belongs to exactly one required recipient `User`** (`Notification.recipientUserId`), never a split staff/client identity path. Blueprint Section 14.9 itself names a single "User (staff or client)" target; unlike `Message`/`ConversationParticipant`, whose staff/client identity split exists to record differing authorship semantics, a notification recipient requires no such split — every activated client account is already a `User` row (`ClientProfile.userId`).
  - **The recipient relation uses `onDelete: Restrict`**, matching this schema's established convention for a row that documents something about a User (`AuditLog.actorId`, `BookingStatusHistory.changedByUserId`, `PaymentStatusHistory.changedByUserId`) rather than the `onDelete: Cascade` convention reserved for pure account extensions (`Session`, `Account`, `StaffProfile`). This preserves a Notification if its recipient User is later deactivated — consistent with User accounts being deactivated, never deleted, under this model.
  - **`eventKey` is a free-form, stable string**, not a closed enum, because no authoritative source in this repository defines a complete notification-type list — unlike `VisaCaseStatus`, where D-017 resolved an explicitly blueprint-flagged open decision (blueprint Section 13.8/16.2), nothing flags Notification's event types as needing closure. No Notification enum of any kind is introduced by this checkpoint. Any illustrative event-key examples recorded in `schema.prisma`'s doc comments (e.g. `"payment.confirmed"`, `"document.rejected"`) are non-exhaustive and not reserved by this decision — the future notification-producing service defines the real set.
  - **Durable display content is `title` and `message`**, the minimum payload an in-app alert needs to render itself, both required (non-nullable) strings.
  - **An optional `actionPath` is an internal relative application path**, never a permanent public URL or an arbitrary external URL, mirroring `MessageAttachment.storageKey`'s established never-a-public-URL convention.
  - **No generic `resourceType`/`resourceId` pair and no JSON `metadata` field are added.** Unlike `AuditLog.entityType`/`entityId` (accepted there for forensic record-keeping, not a live client-facing dereference target), a polymorphic reference pair or open JSON field on a user-facing Notification would both lose referential integrity and risk becoming a place sensitive payment, document, visa, or conversation content gets copied in.
  - **Read state is `readAt DateTime?`**, not a mutable boolean or a separate read-history model, mirroring `ConversationParticipant.lastReadAt`'s established pattern. Notification has exactly one recipient (unlike Conversation's many-participants problem that required `lastReadAt` on its own join model), so a single nullable column on the Notification row itself is sufficient.
  - **Lifecycle fields are `createdAt` and `readAt` only** — no `updatedAt`, `expiresAt`, or `archivedAt`. No `updatedAt` is added because the one meaningful mutation this row can receive under this foundation — being marked read — already has its own dedicated, more precise timestamp (`readAt`); a generic `updatedAt` would duplicate rather than add information. No `expiresAt`/`archivedAt` is added because no retention or expiry policy is approved anywhere in this repository yet (D-003/D-004 remain Open).
  - **No deduplication/idempotency field is added in Stage A.** Unlike the Payment-domain idempotency keys (D-019), added because specific financially-sensitive writes had no natural uniqueness to retry against, no Notification-producing service, event-production rule, or background job exists yet to design a real retry/duplicate shape against — a dedup field now would be guessing at an undesigned need. This may be added additively once Phase 5's actual event-production and retry design is known.
  - **Notification represents the in-app alert only.** Email delivery (blueprint Section 10.3; ADR-001's email-notification boundary) remains a separate, later concern — no email provider, channel, sent timestamp, delivery-status, or delivery-attempt field is added. The portal remains the system of record; email is an outbound side effect, not modeled by this table.
  - **Indexes are `@@index([recipientUserId, createdAt])`** (a recipient's inbox ordered by creation time) **and `@@index([recipientUserId, readAt, createdAt])`** (filtering a recipient's unread notifications together with creation-time ordering).
  - **No hand-written PostgreSQL CHECK constraint or partial index is specified for this minimal shape.** Unlike every earlier schema-foundation checkpoint in this file (D-016 through D-019), each of which required at least one hand-written constraint because of nullable-FK exclusivity or a conditional-string requirement, this checkpoint's shape has neither — this is the first schema-foundation checkpoint requiring no migration hand-augmentation.
  - **Future service-layer invariants** — none enforced by this schema, all left to the future notification-producing/consuming service: non-blank `eventKey`/`title`/`message`; that `actionPath`, when present, is a genuine internal relative path; data minimization (title/message must never contain sensitive payment, document, visa, or private conversation content, nor any credential, token, or secret); recipient-scoped authorization on every read; and which platform event maps to which recipient(s).
  - **No migration has been generated or applied during Stage A.** No `prisma migrate` command was run. `schema.prisma`'s new `Notification` model and `User.notificationsReceived` relation do not correspond to any database table, column, constraint, or index anywhere — local, staging, or production — until a later stage generates and a still-later stage applies the migration.
- **Rationale:** Blueprint Section 14.9 gives Notification only a one-sentence description with no field list, type list, or delivery model — every field beyond the single required recipient relation is therefore a design inference, not a blueprint-stated requirement, and this entry records each inference's justification explicitly rather than presenting it as settled fact (mirroring D-017's and D-018's own discipline for VisaCase and Document, each of which had similarly thin blueprint text). The single required-`User`-recipient shape, `Restrict`-on-delete, free-form `eventKey`, and nullable-timestamp read state each directly reuse an already-established pattern elsewhere in this schema (`ClientProfile.userId`, `AuditLog`/`BookingStatusHistory`/`PaymentStatusHistory`'s actor-relation convention, `Document.category`'s free-text precedent, and `ConversationParticipant.lastReadAt`, respectively) rather than inventing a new one, consistent with this schema's established caution against premature, likely-to-be-reworked design (the same reasoning D-017 applied to VisaRequirement and D-019 applied to PaymentAdjustment). Omitting `resourceType`/`resourceId`, JSON `metadata`, deduplication, and every email-delivery field follows the same discipline in the opposite direction: each is a plausible future need with no current approved shape to build against, so none is guessed at now.
- **Constraint:** No hand-written PostgreSQL CHECK constraint or partial unique index is required by this checkpoint's minimal shape — a departure from D-016/D-017/D-018/D-019, each of which specified at least one for a future migration-generating stage. Should a future stage introduce any conditional or cross-column rule (e.g., a dedup key with conditional uniqueness), that constraint must be decided and documented in its own later entry, not silently added to a migration. Nothing in this entry authorizes generating or applying a migration; that remains a separate, later, explicitly authorized stage.
- **Deferred:** Notification repositories and services; event-production rules (which platform events produce a notification, and for which recipients); whether creation happens synchronously in the same transaction as its triggering write or via a background job; retry and deduplication behavior; API routes; admin dashboard and client portal notification UI; recipient-scoped authorization enforcement on every fetch; list and mark-read behavior; email delivery and delivery-attempt tracking; background workers; notification preferences and opt-out rules; expiry, archival, deletion, and retention rules; and tests and seed data. All remain future implementation work, most falling under Phase 5 ("notifications," docs/HERITAGE_V3_TASK_BOARD.md).
- **Effect:** Implements Stage A (schema-source and decision documentation only) of the Notification Schema Foundation slice of the Phase 1 "Implement database schema and reviewed migrations from the approved Phase 0 entity plan" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. That checklist item remains unchecked during Stage A: no reviewed Notification migration has been generated or applied to any environment. After a later reviewed migration is successfully generated, hand-reviewed, and applied to local `heritage_v3_dev` (and verified, mirroring D-017/D-018/D-019's own Stage B/C verification pattern), Notification will resolve the final unresolved entity-plan blocker D-019's implementation-status update identified — but that verification has not occurred as part of this entry. `docs/HERITAGE_V3_TASK_BOARD.md` is not updated by this entry; its update belongs to its own later, separately authorized stage, exactly mirroring D-017/D-018/D-019's own discipline. This entry affects only the "Implement database schema and reviewed migrations" checklist line specifically — it does not affect, and makes no claim about, the separate staff account/role/assignment management checklist item, the not-yet-verified real-PostgreSQL sign-in note, the deployment-pipeline-skeleton checklist item, or overall Phase 1 completion-gate status, all of which remain independently unresolved. `ActivityLog` remains deferred to Phase 4 exactly as D-018 already documented, specifically for document access/download logging (blueprint Section 12.3) — this entry does not broaden, reinterpret, or supersede D-018's ActivityLog decision, and does not claim ActivityLog is implemented anywhere. Notification is the final unresolved entity-plan blocker specifically according to D-019's own recorded coverage assessment (Permission/D-012, VisaRequirement/D-017-D-018, ActivityLog's access/download-logging purpose/D-018, TourPackage/ItineraryVersion/the Booking Schema Foundation checkpoint, and PaymentAdjustment/D-019 are each already covered by their own documented deferral).
