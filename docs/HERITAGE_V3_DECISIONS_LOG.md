## D-011 — Authentication library selection

- **Status:** Accepted
- **Date accepted:** July 10, 2026
- **Decision:** Better Auth, with its official Prisma/PostgreSQL adapter (`better-auth/adapters/prisma`), is the authentication and session library for Heritage Philippines V3. This fulfills ADR-001's deferred "Select a vetted authentication/security library after checking its official documentation" follow-up action (ADR-001, "Immediate, pre-Phase-1-scaffolding").
- **Rationale:** Evaluated against Lucia and Auth.js (NextAuth). Lucia was deprecated by its maintainer in March 2025 and is no longer distributed as an installable library (repositioned as an educational resource) — disqualified as "maintained." Auth.js's Credentials provider is architecturally hardcoded to JWT-only sessions and cannot use real database-backed sessions without unsupported workarounds, conflicting with ADR-001's stated preference for "database-backed session state" and this project's rule against inventing custom session mechanics. Better Auth supports credentials-based login and database-backed sessions natively through its official Prisma adapter, hashes passwords internally (scrypt) rather than requiring custom cryptography, and supports disabling public self-service sign-up (`emailAndPassword.disableSignUp`) while keeping sign-in enabled — required to satisfy blueprint Section 7's invitation-only-signup rule. As of September 2025 the Auth.js/NextAuth maintainers themselves joined the Better Auth project and now recommend it for new projects. It requires no serverless-specific hosting platform, remaining compatible with Hostinger-managed VPS-class Node.js hosting (ADR-001).
- **Constraint:** This selects a library within ADR-001's already-approved architecture (Next.js, Prisma, PostgreSQL, cookie-based sessions) and does not alter the accepted technology stack. It does not implement custom cryptography, password hashing, or session-token primitives, consistent with ADR-001's security requirements.
- **Effect:** Implements the technology choice underlying the Phase 1 "Establish authentication/session foundation" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. Final MFA/SSO and any additional auth providers remain open per blueprint Section 16.2.

## D-012 — Better Auth admin plugin evaluated and rejected for staff management

- **Status:** Accepted
- **Date accepted:** July 13, 2026
- **Decision:** The installed Better Auth admin plugin (`better-auth@1.6.23`, `better-auth/plugins/admin`) was evaluated as a candidate implementation for the Phase 1 "staff account, role, and assignment management" checklist item and rejected. Heritage Philippines V3 will instead implement a custom, audited staff-management service (Approach B) built on the project's existing Prisma client, role guards (`src/lib/auth/roles.ts`, `authorize.ts`, `guards.ts`), and only the non-plugin Better Auth server primitives already in use (e.g., `hashPassword` from `better-auth/crypto`, as `prisma/seed.ts` already does). The `admin` plugin must not be registered in `src/lib/auth/auth.ts`, and no `/api/auth/admin/*` route may exist.
- **Rationale:** Direct inspection of the installed package's compiled source (`dist/plugins/admin/{admin,routes,schema,types,has-permission}.mjs`) found three conflicts with Heritage's requirements: (1) the plugin's endpoints (`set-role`, `create-user`, `ban-user`, `remove-user`, etc.) are unconditionally mounted on the same catch-all handler this project's `/api/auth/[...all]` route already forwards everything to — there is no supported way to use the plugin's server APIs internally without also exposing those endpoints over HTTP, authorized only by the plugin's own parallel `adminRoles`/`ac` permission config rather than this project's fixed six-role model; (2) the plugin redefines `user.role` as an unconstrained string (`type: "string"`, not required), a different shape from this project's own six-value literal-union `role` field, and its `setRole` endpoint has no self-action or "last remaining admin" protection; (3) its mutation endpoints call Better Auth's internal adapter directly with no exposed transaction hook, so an `AuditLog` write can never be atomic with the mutation it records, and `removeUser` is a hard delete rather than the deactivate/reactivate model the blueprint calls for.
- **Constraint:** Heritage retains its fixed, blueprint-approved six-role `Role` enum and keeps permissions expressed in code rather than adopting the plugin's own role/permission system. The staff-management service (implemented in a later checkpoint) must perform staff account creation, role changes, and deactivation/reactivation as direct, atomic Prisma operations (mutation + `AuditLog` insert in one transaction), not through any Better Auth admin-plugin endpoint.
- **Effect:** Closes the Approach A vs. B vs. C evaluation for this checklist item in favor of Approach B. Direct credential provisioning (a System Administrator setting a new staff account's initial password) must be covered by an integration test in the staff-management-service checkpoint that verifies the created account can actually authenticate through the normal Better Auth sign-in path — not merely that a database row was written correctly.

## D-013 — Server-generated booking reference format

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `Booking.bookingReference` (blueprint Section 13.4) is generated exclusively by the server as the literal prefix `HPB-` followed by 20 uppercase hexadecimal characters (e.g. `HPB-7A31F09C45D2E86B104F`), derived from 10 cryptographically secure random bytes via Node's built-in `crypto.randomBytes` — no new dependency is introduced. It is never supplied or overridable by a client: `POST /api/bookings`'s request schema (`apps/web/src/features/bookings/schemas.ts`) has no `bookingReference` field at all, and the service layer (`apps/web/src/features/bookings/service.ts`'s `generateBookingReference`) is the only place this value is produced.
- **Rationale:** Blueprint Section 13.4 names "Booking reference" as a Booking field but specifies no format, sequence, or generation rule anywhere in the blueprint, this decisions log, or the task board — this was an explicitly open item identified during the Booking Schema Foundation checkpoint's implementation reconnaissance. A random, non-sequential, non-year-based identifier avoids leaking booking volume or timing through the reference itself and avoids the coordination cost a sequential counter would require under concurrent creation. `Booking.bookingReference`'s existing database `@unique` constraint (migration `20260716020734_booking_schema_foundation`) remains the final safeguard regardless of generation method; the booking-creation service additionally retries with a freshly generated reference, within a small bounded retry policy, on the rare event of a collision.
- **Constraint:** Sequential or human-readable (e.g., year-prefixed, incrementing) booking references are deferred unless management later requires them — this decision does not preclude revisiting the format, but any change must go through this same decision process, not be introduced ad hoc in a later checkpoint.
- **Effect:** Resolves the previously open "booking reference generation" item identified during the Booking Schema Foundation checkpoint. Implemented as part of the booking workflow enforcement checkpoint (explicit Booking creation, staff-only list/read, initial `BookingStatusHistory`, and the `BOOKING_CREATED` audit entry) — status-transition enforcement and Booking-specific staff-assignment endpoints remain separate, not-yet-implemented checkpoints.

## D-014 — Booking status-transition matrix, authorization, and concurrency

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `PUT /api/bookings/[id]/status` enforces the following fixed, service-layer transition matrix over `BookingStatus` (blueprint Section 13.4's ten values), independent of any database constraint:
  - `DRAFT` → `PENDING_CONFIRMATION`, `CANCELLED`
  - `PENDING_CONFIRMATION` → `CONFIRMED`, `CANCELLED`
  - `CONFIRMED` → `IN_PREPARATION`, `CANCELLED`
  - `IN_PREPARATION` → `DOCUMENTS_REQUIRED`, `VISA_PROCESSING`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `DOCUMENTS_REQUIRED` → `VISA_PROCESSING`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `VISA_PROCESSING` → `DOCUMENTS_REQUIRED`, `READY_FOR_TRAVEL`, `CANCELLED`
  - `READY_FOR_TRAVEL` → `IN_PROGRESS`, `CANCELLED`
  - `IN_PROGRESS` → `COMPLETED`, `CANCELLED`
  - `COMPLETED` → no outgoing transitions (terminal)
  - `CANCELLED` → no outgoing transitions (terminal)

  Authorization mirrors Booking creation/read exactly: `ADMIN_MANAGER` (unconditional) and `TRAVEL_CONSULTANT` (only through the existing active-Client-assignment check, `features/bookings/repository.ts`'s `clientAssignmentFilter` — not a new Booking-specific assignment mechanism, which remains unimplemented). `SYSTEM_ADMINISTRATOR`, `FINANCE_ACCOUNTING`, `VISA_DOCUMENTATION`, and `CLIENT` are not authorized to change Booking status in this checkpoint.

  Concurrency: the caller supplies `expectedStatus` alongside `newStatus`; the service reads the Booking's actual current status inside a `runSerializableWithRetry` transaction and rejects with `BOOKING_CONFLICT` (409, refresh-and-retry message) if the current status no longer matches `expectedStatus` — an optimistic-concurrency check layered on top of (not a replacement for) the transaction's own serializable-conflict retry. A request whose `newStatus` already equals the current status is a no-op (no history row, no audit entry, no write) — required by `booking_status_history_status_changed`'s existing CHECK constraint, which forbids a history row where `previousStatus` equals `newStatus`.

  `DOCUMENTS_REQUIRED` and `VISA_PROCESSING` are treated as coarse overall booking-stage indicators only, not as a model of the detailed, genuinely parallel document/visa workflows blueprint Section 8 describes ("payments, documents, visa processing... may all progress in parallel rather than strictly in sequence"). `Booking.status` remains a single scalar reflecting the booking's overall stage; the future `Document` and `VisaCase` entities (blueprint Sections 14.7, 14.8) own their own detailed, independently-progressing status fields once implemented. No schema change accompanies this decision.
- **Rationale:** Blueprint Section 13.4 lists the ten status values but defines no transition graph or role authority over them, and Section 8 explicitly warns that the underlying workflows are parallel, not sequential — modeling that fully would require the not-yet-implemented `Document`/`VisaCase` entities, which is out of scope for this checkpoint. The matrix above is a deliberately conservative starting point (forward progression plus universal cancellation from any non-terminal state, since no blueprint text restricts when cancellation is allowed) rather than an attempt to fully resolve the parallel-workflow tension now. Reusing active-Client-assignment (rather than introducing Booking-specific `StaffAssignment` targets, which the schema supports but no service/route yet populates) keeps this checkpoint additive and avoids authorizing against an assignment type nothing can currently create.
- **Constraint:** This matrix, and the two-role authorization set, may be revised by a later decision once `Document`/`VisaCase` are implemented and the parallel-workflow question is revisited; until then, no additional role or transition should be added ad hoc outside this decision. A `reason` field for status changes is explicitly deferred, not adopted, in this checkpoint.
- **Effect:** Implements the "Booking status-transition enforcement" checkpoint referenced in `docs/HERITAGE_V3_TASK_BOARD.md`. Booking-specific staff-assignment management and any Document/VisaCase-driven status model remain separate, not-yet-implemented checkpoints.

## D-015 — Booking-specific staff-assignment enforcement

- **Status:** Accepted
- **Date accepted:** July 16, 2026
- **Decision:** `PUT /api/bookings/[id]/assignment` lets an Admin / Manager set or replace the Travel Consultant assignment for a specific Booking, extending the existing `features/assignments` shared implementation (`TargetKind` now includes `'BOOKING'` alongside `'LEAD'`/`'CLIENT'`) rather than introducing a separate Booking-assignment service. Booking assignment is **independent of the underlying Client assignment**: the assignee is never required to already hold the Booking's Client's active assignment, and no cascade exists in either direction — ending, replacing, or creating a Client assignment never touches an existing Booking assignment, and vice versa. This checkpoint **does not affect** Booking read, list, or status-transition authorization: `features/bookings/repository.ts`'s `clientAssignmentFilter` and the D-014 status-transition authorization rules are unchanged and continue to gate Travel Consultant access via active Client assignment only. This checkpoint is **`PUT`-only** — no `DELETE`/removal-without-replacement endpoint exists for Booking assignment; ending a Booking assignment without an immediate replacement is deferred, not decided against permanently.
- **Rationale:** Blueprint Section 4.2 explicitly grants Admin / Manager the capability to "manage staff accounts' operational role assignments and lead/client/booking assignments," naming Booking directly — the same explicit basis Lead and Client assignment already rely on. The underlying `StaffAssignment.bookingId` column, its three-way `leadId`/`clientId`/`bookingId` XOR CHECK constraint, and its partial unique active-Booking-assignment index were already added by the Booking Schema Foundation checkpoint (migration `20260716020734_booking_schema_foundation`) but never used by any service or route until now — extending the existing generic `setAssignment`/`endAssignment` control flow (idempotent same-assignee replay, `REASON_REQUIRED` only when replacing an active assignment, atomic end-old/create-new/audit inside `runSerializableWithRetry`) maximizes reuse of already-reviewed, already-tested logic rather than duplicating it. Leaving Booking read/status authorization on Client-level assignment avoids conflating two separate concerns — "who currently works this Booking" (new, additive) and "who may read or transition this Booking" (existing, unchanged) — until a future decision, if any, deliberately links them.
- **Constraint:** Whether a Booking assignment must eventually require, or should remain independent of, the Client assignment; whether Booking read/status authorization should ever incorporate Booking-level assignment; and whether assignment removal without replacement should be added, are all left open for a future decision — none is silently resolved here. No Prisma schema or migration change was required or made for this checkpoint.
- **Effect:** Implements the "Booking-specific staff-assignment enforcement" checkpoint referenced in `docs/HERITAGE_V3_TASK_BOARD.md`. `features/bookings/repository.ts`, its status-transition service, and their routes remain completely unchanged.

## D-016 — Conversation Schema Foundation: schema shape, read-state model, and migration-history remediation

- **Status:** Accepted
- **Date accepted:** July 17, 2026
- **Decision:** The Conversation Schema Foundation schema-only checkpoint (blueprint Sections 4.2–4.6, 9.1, 10, 10.1, 10.2, 10.3, 13.9, 14.3) adds `ConversationCategory` (the eight closed categories in Section 10.2, adopted directly), `MessageVisibility` (`CLIENT_VISIBLE`/`INTERNAL_NOTE`, Section 10.1), `ConversationParticipantRole` (`CLIENT`/`TRAVEL_CONSULTANT`/`FINANCE_ACCOUNTING`/`VISA_DOCUMENTATION` — deliberately excluding `ADMIN_MANAGER`, since Section 10.1 treats Admin/Manager's full visibility as a role-wide grant, not thread participation), `Conversation`, `ConversationParticipant`, `Message`, and `MessageAttachment`. Key shape decisions:
  - `Conversation.clientId` is a required direct FK to `Client` (mirroring `Lead`/`Proposal`/`Booking`), since a Conversation may have no related Booking or Proposal at all.
  - A Conversation may reference **at most one** related record in this checkpoint — an optional `relatedBookingId` or an optional `relatedProposalId`, never both (enforced by CHECK, see below). A `relatedVisaCaseId` is not modeled: VisaCase does not exist anywhere in this schema yet, matching its existing deferral as a `StaffAssignment` target.
  - `ConversationParticipant` uses two mutually exclusive identity paths — `staffUserId` (User) for `TRAVEL_CONSULTANT`/`FINANCE_ACCOUNTING`/`VISA_DOCUMENTATION` and `clientProfileId` (ClientProfile) for `CLIENT` — tied together by one CHECK constraint per participant row. `joinedAt`/`removedAt` preserve membership history (mirroring `StaffAssignment.endedAt`) rather than deleting a row on removal, and a person may rejoin after removal.
  - Read state (Section 10.1's "read status") is modeled as **`ConversationParticipant.lastReadAt`** — a per-participant timestamp, not a shared boolean on `Message` — because a Conversation may have more than two participants and a single flag cannot represent "read" correctly for all of them at once. A `Message` is considered read for a given participant when the message's `createdAt` is at or before that participant's `lastReadAt`; the future service layer owns updating `lastReadAt` and resolving timestamp/ordering edge cases. `lastReadAt` is read-state storage only — it never itself grants or implies authorization.
  - `Message` uses the same two-path author split (`authorStaffUserId`/`authorClientProfileId`), and this checkpoint adds one content-level database restriction beyond path exclusivity: a client-authored message can never have `visibility = INTERNAL_NOTE`.
  - `MessageAttachment` stores only the minimal metadata the backend-mediated storage boundary needs (`storageKey` unique, `fileName`, `mimeType`, `fileSizeBytes`) — never a public URL.
  - No Conversation status/lifecycle field, subject/title field, or `lastMessageAt` denormalization is added — none is defined anywhere in blueprint Section 10.
  - Conversation/message **authorization**, and the cross-table client-consistency invariants (`relatedBooking.clientId`/`relatedProposal.clientId` must equal `Conversation.clientId`; a `CLIENT` participant must belong to `Conversation.clientId`), are explicitly **not implemented** in this checkpoint. A future enforcement checkpoint must account for all of: Admin/Manager's full, unconditional visibility into every Conversation; a CLIENT actor's ownership of the Conversation's Client (via ClientProfile) together with explicit `ConversationParticipant` membership; a Travel Consultant's active Client/Booking assignment; and explicit `ConversationParticipant` membership for Finance/Visa Documentation actors, who have no assignment-based path onto a Conversation at all. Conversation authorization must never be described or implemented as "participant rows alone." Server-side exclusion of `INTERNAL_NOTE` messages from every client-facing query is likewise deferred to that same future checkpoint.
- **Rationale:** Directly follows Section 10.1's documented structure (thread/participants/messages/attachments, client-visible-vs-internal-note) and Section 10.2's closed category list, without inventing any undocumented field (status, title, `lastMessageAt`). The per-participant `lastReadAt` model was chosen over a shared Message-level boolean specifically because Section 10.1 describes more than two possible participants per thread (Travel Consultant, optional Finance, optional Visa Documentation, plus the Client), which a shared flag cannot represent correctly. The staff/client split-identity pattern on both `ConversationParticipant` and `Message` reuses this schema's established `StaffAssignment`/`ProposalAcceptance` precedent rather than inventing a new attribution shape.
- **Migration-history note:** This checkpoint's schema is split across two migrations rather than one, because of an implementation-session incident: `prisma migrate dev --create-only` was run a second time to confirm the first run's exit code, and Prisma's documented behavior of always applying any already-pending unapplied migration at the start of a `migrate dev` invocation (regardless of `--create-only` on that specific run) caused the first-generated migration — at that point containing only the auto-generated tables/enums/standard indexes/foreign keys, before any hand-written constraint had been added — to be applied to the local `heritage_v3_dev` database. Inspection confirmed: all four new tables were empty at the time (no data was ever at risk), no object outside the four new tables referenced them, and no pre-existing table's structure was altered. Rather than editing the already-applied migration file after the fact (which would desync it from the checksum Prisma already recorded against it in `_prisma_migrations`, and `prisma migrate resolve --rolled-back` is not a valid operation against a migration that completed successfully — it did not fail), the remediation was forward-only: migration `20260717010445_conversation_schema_foundation` (tables/enums/standard indexes/FKs) was left byte-for-byte unchanged; the second, previously-empty migration was renamed to `20260717010520_conversation_schema_constraints`, populated with every hand-written PostgreSQL CHECK constraint and partial unique index this checkpoint requires, and reviewed. Following that review, **both migrations were successfully applied to local `heritage_v3_dev` on July 17, 2026** via `prisma migrate dev` (the base migration was already applied from the incident above; only the constraints migration was newly applied in this step). `prisma migrate status` subsequently reported all 8 migrations applied and the schema up to date. A read-only PostgreSQL catalog check confirmed all four hand-written CHECK constraints (`conversation_related_record_exclusive`, `conversation_participant_identity_role_match`, `message_author_path_exclusive`, `message_client_author_not_internal_note`) are present and validated, and both partial unique indexes (`conversation_participant_active_staff_key`, `conversation_participant_active_client_profile_key`) are present, unique, and valid with the correct predicates. This application is confirmed for local `heritage_v3_dev` only — application to staging or production has not occurred as part of this checkpoint and must be verified independently before either environment is considered current.
- **Constraint:** No later checkpoint should combine these two migrations or edit either one's SQL after the fact. Conversation/message authorization, cross-client consistency checks, VisaCase as a third related-record type, email notification delivery (Section 10.3), and attachment malware scanning/signed-URL issuance (Section 12.3) all remain open for a future checkpoint, not resolved here. Conversation-creation authorization (who may open a Conversation, and whether a client may self-initiate one for particular categories) and the exact policy governing who may author a Message (CLIENT ownership plus active participation; explicit active participation for Finance/Visa Documentation; Travel Consultant assignment with the participant-row question still open; and whether Admin/Manager may post by role alone without participant membership) are both likewise left open — authorship foreign keys alone must never be treated as proof of authorization by any future implementation.
- **Effect:** Implements the Conversation Schema Foundation slice of the Phase 1 "Implement database schema and reviewed migrations from the approved Phase 0 entity plan" checklist item in `docs/HERITAGE_V3_TASK_BOARD.md`. Payment, Document, and VisaCase schema remain unimplemented, so that checklist item is not marked complete by this decision.

## D-017 — Visa Case Schema Foundation: MVP granularity, status list, and Stage A schema shape

- **Status:** Accepted
- **Date accepted:** July 17, 2026
- **Decision:** The Visa Case Schema Foundation checkpoint (blueprint Sections 4.5, 4.7, 5, 5.1, 5.2, 8, 13.8, 14.1, 14.8) is implemented in stages; this entry covers **Stage A (schema-and-decision-documentation only — `schema.prisma` and this decisions-log entry, with no migration created or applied)**. Approved:
  - **One VisaCase per Booking for the MVP.** A Booking may have zero or one VisaCase; a VisaCase belongs to exactly one Booking. Enforced by `VisaCase.bookingId String @unique` — a plain unique constraint, not a partial one, since this checkpoint has no historical/superseded VisaCase concept for a partial index to distinguish. Multiple historical or simultaneous VisaCases per Booking, and traveler-level VisaCase modeling (one case per traveler on a multi-traveler Booking), are both explicitly out of scope.
  - **The exact nine-value `VisaCaseStatus` enum:** `OPENED`, `REQUIREMENTS_PENDING`, `UNDER_REVIEW`, `ADDITIONAL_INFORMATION_REQUIRED`, `READY_FOR_SUBMISSION`, `SUBMITTED`, `APPROVED`, `REJECTED`, `WITHDRAWN`, with `APPROVED`, `REJECTED`, and `WITHDRAWN` as terminal. This is the one status enum in this schema whose values are set by explicit decision rather than adopted directly from blueprint text — blueprint Section 13.8 states "the detailed visa status list will be finalized alongside implementation planning" and Section 16.2 separately lists it as an open decision; this entry is that finalization. No transition matrix is adopted or implied by the enum's declaration order or by which values are marked terminal — transition enforcement (including enforcing terminality) is explicitly deferred to a later service/API checkpoint, mirroring how `BookingStatus`'s transition matrix was deferred to its own later, separate D-014 checkpoint.
  - **Direct `Client` and `Booking` relations on `VisaCase`**, mirroring this schema's established `Lead.clientId`/`Proposal.clientId`/`Booking.clientId`/`Conversation.clientId` convention, both `onDelete: Restrict`. `VisaCase.clientId` must always equal the Client reachable through `VisaCase.booking.clientId` — a cross-record rule Prisma cannot enforce, deferred to mandatory service-layer verification at write time, the same way `Booking.clientId` vs. its `ProposalVersion.proposal.clientId` is already deferred. No duplicated or derived-column schema logic was used to attempt to solve this.
  - **VisaCase as a StaffAssignment target at the schema level only.** `StaffAssignment.visaCaseId` (nullable) is added in Stage A; the existing three-target XOR CHECK constraint and the corresponding partial unique active-assignment index are **not yet altered in any database** — a later migration-generating stage must extend `staff_assignment_lead_xor_client_xor_booking` to a four-target `staff_assignment_lead_xor_client_xor_booking_xor_visa_case` (`CHECK (num_nonnulls("leadId", "clientId", "bookingId", "visaCaseId") = 1)`) and add `staff_assignment_active_visa_case_key` (`WHERE "endedAt" IS NULL AND "visaCaseId" IS NOT NULL`). `features/assignments`' `TargetKind` union, service, audit, and authorization code are untouched in Stage A and remain so until a separate, later assignment-enforcement checkpoint, mirroring D-015's precedent for Booking exactly.
  - **VisaCase as the third optional Conversation related-record target at the schema level only.** `Conversation.relatedVisaCaseId` (nullable) is added in Stage A; the existing two-column `conversation_related_record_exclusive` CHECK is **not yet altered in any database** — a later migration-generating stage must replace it with `CHECK (num_nonnulls("relatedBookingId", "relatedProposalId", "relatedVisaCaseId") <= 1)`.
  - **Every `VisaStatusHistory` row requires an attributable staff actor under D-017's MVP policy.** `changedByUserId` is required, not nullable, matching `BookingStatusHistory` rather than `LeadStatusHistory`. Blueprint Section 5.1 permits deliberate creation through an explicit staff or system action; D-017 deliberately narrows the current MVP schema to staff-attributed creation and status history. A future automated or actorless VisaCase transition would require a superseding decision and schema change.
  - **`VisaRequirement` and all Document entities (`Document`, `DocumentRequirement`, `DocumentReview`) are deferred**, not modeled even as a placeholder. `VisaRequirement`'s one-sentence blueprint description (Section 14.8: "defines what is required for a specific VisaCase (documents, forms, deadlines)") overlaps structurally with `DocumentRequirement` (Section 14.7: "defines which document categories are required for a given Booking or VisaCase"), and neither `Document` nor `DocumentRequirement` exists anywhere in this schema yet.
  - **Assignment-service wiring, VisaCase authorization, the status-transition matrix, API routes, admin dashboard and client portal UI, and notifications are all deferred** to later, separate checkpoints — none is implemented, implied, or assumed by Stage A's schema shape.
  - **No migration has been generated or applied during Stage A.** No `prisma migrate` command was run. `schema.prisma`'s new and changed models, fields, relations, and doc comments do not correspond to any database table, column, constraint, or index anywhere — local, staging, or production — until a later stage generates and a still-later stage applies the migration.
- **Rationale:** Blueprint Section 5.2's conceptual diagram presents VisaCase as a singular conditional relationship (`Booking --(conditional, only if required)--> Visa Case`) and Section 8 confirms a visa case is conditional and never automatic, but neither section independently defines complete cardinality — the blueprint does not itself state whether a Booking could ever have more than one VisaCase over time. D-017 is the explicit decision establishing zero or one VisaCase per Booking for the MVP, chosen because it avoids adding an unapproved Traveler entity or storing loosely modeled applicant identity data during the MVP. The plain-unique (not partial-unique) `bookingId` constraint reflects that this checkpoint has no "current vs. superseded" VisaCase concept to distinguish, unlike `ProposalVersion` or `StaffAssignment`. Deferring `VisaRequirement` and Document entities avoids prematurely designing overlapping structures before the document domain (`Document`/`DocumentRequirement`) is implemented, consistent with this schema's established caution about not modeling a shape that would likely need reworking (e.g. TourPackage deferred entirely). Staging the checkpoint (schema-and-decision-documentation-only Stage A, migration-generation Stage B, application in a still-later stage) is a direct, deliberate response to the D-016 migration-history incident: keeping schema editing, migration generation, and migration application as separate, individually-reviewable steps reduces the chance of a repeat, and Stage A explicitly documents that Stage B must invoke `prisma migrate dev --create-only` exactly once.
- **Constraint:** The four-target `StaffAssignment` XOR CHECK, its `visaCaseId` partial unique index, the three-target `Conversation` related-record CHECK, and the `visa_status_history_status_changed` CHECK are all specified here as requirements for the future migration-generating stage but must not be added to any migration file during Stage A. Who may create or change a VisaCase (beyond Visa Documentation Staff, blueprint Section 4.5), whether Admin/Manager's Section 4.2 "full visibility" extends to VisaCase edit capability, and how VisaCase-level and Booking-level `StaffAssignment` should interact are all left open, not resolved here.
